## 캐시로 최적화

### 캐시 저장 종류

브라우저가 사용 빈도, 파일 크기에 따라 알아서 지정함. 커스텀할 수 없음

- 메모리 캐시 : RAM에 저장
- 디스크 캐시 : 파일 형태로 디스크에 저장

### 캐시확인

- Network 패널의 Disable cache 설정 해제해야 확인 가능
    - 단순 새로고침 후 확인하면 memory cache가 많음 : 구글의 리소스가 메모리에 캐시돼서
    - 브라우저 완전 종료 후 최초 접속이 해당 사이트인 경우 disk cache가 많음 : 브라우저가 종료될 때 메모리를 비우면서 파일 형태로 저장하기 때문

### Cache-Control

- 서버에서 보낸 응답 헤더의 `Cache-Control` 이 있어야 캐시 저장됨
- 옵션
    - `no-cache` : 캐시를 사용하기 전 서버에 검사 후 사용
    - `no-store` : 캐시 사용 안 함
    - `max-age` : 캐시의 유효 시간 (단위 : 초) `max-age=60`
        
        `max-age=0` = `no-cache`
        
        - `public` : 모든 환경에서 캐시 사용 가능
        - `private` : 브라우저 환경에서만 캐시 사용. 외부 캐시 서버(중간서버 등)에서는 사용 불가
- 캐시된 리소스와 서버 최신 리소스의 비교 방법 : Etag값 사용
    - 캐시 유효 시간이 만료되면 캐시 리소스를 계속 사용할지, 아니면 갱신해야 할지 서버에 확인을 요청
    - 이때 캐시 리소스의 응답 헤더에 있던 Etag값을 비교해서 최신 리소스와 비교
- 파일별 캐시 설정
    - HTML은 no-cache설정이 일반적 : 항상 최신 버전의 웹 서비스를 사용하기 위해
    - js, css는 빌드하면서 파일명에 해시값이 포함되기 때문에 최신 파일명이 포함된 HTML만 불러왔다면 캐시 필요없이 최신화됨
